Bedrock

An opinionated observability framework for Go. Unified tracing, metrics, profiling, and structured logging — no globals, everything flows through context.Context.

View on GitHub pkg.go.dev

Installation

go get github.com/kzs0/bedrock

Features

Automatic Metrics

Every operation automatically records count, success, failure, and duration measurements.

Controlled Cardinality

Define metric labels upfront to prevent explosion of label combinations in production.

W3C Trace Context

Standards-compliant distributed tracing with automatic header propagation across services.

HTTP Instrumentation

Built-in middleware for request handling and instrumented clients for outbound calls.

Observability Server

Endpoints for Prometheus metrics, pprof profiling, and health checks out of the box.

Production Defaults

Security timeouts and DoS protections like Slowloris mitigation via ReadHeaderTimeout.

Quick Start

Initialize Bedrock at application startup. The returned context carries all observability state and should be propagated throughout your application.

package main import ( "context" "net/http" "github.com/kzs0/bedrock" ) func main() { // Initialize bedrock with default configuration ctx, close := bedrock.Init(context.Background()) defer close() // Create an instrumented HTTP handler mux := http.NewServeMux() mux.HandleFunc("/api/users", handleUsers) // Wrap with bedrock middleware handler := bedrock.HTTPMiddleware(ctx, mux) http.ListenAndServe(":8080", handler) }

Core Concepts

Bedrock organizes work into a hierarchy of observable units. This structure provides flexibility in how you instrument your code while maintaining consistent telemetry.

Operation

Major units of work that generate full metrics: count, success, failure, and duration. Operations default to success; failures are explicitly registered via error attributes.

Source

Long-running processes that can spawn child operations and track aggregates. Use sources for background workers, queue consumers, or persistent connections.

Step

Lightweight tracing spans within operations without separate metrics. Use steps to add granularity to traces without increasing metric cardinality.

Operations

Operations are the primary unit of work in Bedrock. Each operation automatically records metrics and creates a trace span.

func handleUsers(w http.ResponseWriter, r *http.Request) { ctx := r.Context() // Start an operation - automatically tracks duration and success/failure op := bedrock.Operation(ctx, "fetch_users") defer op.Done() users, err := db.GetUsers(ctx) if err != nil { // Register the error - marks operation as failed op.Register("error", err.Error()) http.Error(w, "internal error", 500) return } // Add attributes for debugging op.Register("user_count", len(users)) json.NewEncoder(w).Encode(users) }

Configuration

Configure Bedrock via options or environment variables.

// Via code ctx, close := bedrock.Init(ctx, bedrock.WithService("my-service"), bedrock.WithLogLevel("debug"), ) // Or via environment variables // BEDROCK_SERVICE=my-service // BEDROCK_LOG_LEVEL=debug
